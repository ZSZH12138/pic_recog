# 🚀 RK3588 视频流内容安全审核架构方案 (V2.0 完整版)

## 一、 模型与算法矩阵库 (全量量化至 INT8)

所有模型必须通过 `rknn-toolkit2` 转换为 RKNN 格式并开启 INT8 量化，模型总内存占用控制在 50MB 以内。

1.  **粗分引擎 (漏斗第一级):**
    * `YOLOv8n-cls`: 多标签图像分类。输出：`正常`、`疑似暴恐/旗帜`、`疑似敏感人物`。
2.  **视觉检测引擎 (漏斗第二级):**
    * `YOLOv8n-det`: 暴恐场景检测。
    * `YOLOv8n-det`: 非法旗帜检测。
    * `YOLOv8n-face`: 专门的高精度人脸坐标检测。
3.  **身份特征引擎 (新增):**
    * `MobileFaceNet`: 接收人脸小图，输出 512 维特征向量 (Float32)。
4.  **文字审核引擎 (新增):**
    * `PP-OCRv4-mobile (Det + Rec)`: 轻量级文字检测与识别。
    * `AC自动机 (C++ 算法)`: 敏感词典多模式极速匹配。

## 二、 硬件算力与内存分配策略

| 硬件模块 | 负责任务 | 调度策略 |
| :--- | :--- | :--- |
| **NPU Core 0** | 100% 跑 `YOLOv8n-cls` (粗分) | **独占绑定**，保障基础流水线的绝对高并发和低延迟。 |
| **NPU Core 1 & 2** | 跑细分检测、FaceNet、PP-OCR | **资源池共享**。由粗分报警或时间触发，NPU 驱动自动分配空闲核心。 |
| **RGA 加速器** | 图片 Resize、Crop、色彩空间转换 | **Zero-CPU 消耗**。负责将不同尺寸图片秒切为 640x640 或 112x112。 |
| **C++ 内存池** | 承接 Node.js 视频帧、存放黑名单向量 | 预分配固定大小（如 300MB）。避免 V8 引擎 GC 引起的内存抖动。 |
| **CPU (ARM NEON)** | 向量余弦相似度计算、AC 自动机查词 | 利用 SIMD 指令集加速，几千人的向量比对在 1 毫秒内完成。 |

## 三、 全链路异步流水线 (核心执行流)

整体管线分为三条并行的处理流。C++ 内存池中的每一个 Block 仅在确认安全后释放，确保极低的内存占用。

### 📌 1. 主干极速流 (针对 100% 视频帧)
* **入站:** Node.js 接收到视频抽帧，直接写入 C++ 环形内存池，获取指针 `ptr`。
* **预处理:** 调用 RGA 将原图缩放至 640x640。
* **粗筛:** Core 0 (`YOLOv8n-cls`) 读取张量。
    * *正常:* 立即释放 `ptr`，返回安全。**(80% 流量在此终结)**
    * *疑似暴恐/旗帜:* 内存锁定，触发【视觉精细流】。
    * *疑似敏感脸:* 内存锁定，触发【人脸判定流】。

### 📌 2. 视觉精细流 & 人脸判定流 (按需触发，Core 1/2 并发)
* **暴恐/旗帜:** Core 1/2 跑 `YOLOv8n-det`，输出确切的违规 Bounding Box，随后释放内存并上报 Node.js。
* **人脸比对:**
    1.  Core 1/2 (`YOLOv8n-face`) 提取人脸坐标。
    2.  RGA 硬件瞬间裁剪出人脸并 Resize 到 112x112。
    3.  Core 1/2 (`MobileFaceNet`) 提取 512 维特征向量。
    4.  C++ 层 CPU (NEON) 遍历常驻内存的 `std::vector` (几千人黑名单，约 10MB)，计算余弦相似度。
    5.  命中阈值 (如 > 0.85)，释放内存并上报 Node.js 人物 ID。

### 📌 3. OCR 降频审核流 (独立时钟触发，如每 1-2 秒 1 帧)
* **抽帧分流:** Node.js 控制时间戳，每秒抽 1 帧的指针 `ptr` 独立送入文本流水线。
* **文字提取:** Core 1/2 先后执行 PP-OCR 的 Det (找文本框) 和 Rec (认字)。
* **词库匹配:** 提取的字符串在 C++ 层过一遍 AC 自动机（支持几万个自定义不良词汇）。
* **命中处置:** 若含有敏感词，释放内存并上报 Node.js 违规词汇及位置。

## 四、 系统扩展与前端对接规范

1.  **黑名单热更新 (Zero Downtime):**
    当需要增加劣迹艺人时，系统无需重启。Node.js 提取其照片的特征向量，通过 N-API 调用 C++ 的 `AddVector(id, feature)` 接口，直接将数据追加到内存的向量数组末尾即可实时生效。
2.  **前端渲染解耦:**
    在最终的展示结果呈现上，C++ 层和后端不需要做任何图像的重新绘制（避免内存二次拷贝）。后端通过 JSON 接口返回违规类型、文本内容以及坐标数组 `[x1, y1, x2, y2]`。前端获取数据后，利用 Canvas API 的 `strokeRect` 等方法，在原视频流或图片上方直接渲染红框和警告信息，确保了极高的性能和前后端的分离。

这套 V2.0 架构在不增加额外物理内存和中间件依赖的前提下，将 RK3588 的视觉和并发能力压榨到了极致，并且为未来的模型扩展预留了充足的代码接口。
